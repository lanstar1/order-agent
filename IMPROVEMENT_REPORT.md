## 1. 보안 취약점 (긴급)

### 1.1 API 키 평문 노출
- `.env` 파일에 ANTHROPIC_API_KEY, GOOGLE_API_KEY, ERP_API_KEY가 평문 저장
- Git 히스토리에 이미 커밋되었을 가능성 있음
- **조치:** `.gitignore`에 `.env` 추가, 노출된 키 전부 재생성, `git-filter-branch` 또는 BFG로 히스토리 정리

### 1.2 CORS 전면 개방
- `main.py` 라인 44-49: `allow_origins=["*"]` → 모든 도메인에서 API 호출 가능
- **조치:** 실제 서비스 도메인만 허용 (`https://your-render-domain.onrender.com`)

### 1.3 인증 미비
- 대부분의 API에 인증 없이 접근 가능 (주문처리, 고객정보, 단가표 등)
- 로그인은 있으나 JWT/세션 검증 미적용
- **조치:** FastAPI `Depends()`로 Bearer Token 검증 미들웨어 추가

### 1.4 약한 비밀번호 해싱
- `auth.py`: SHA256 사용 (salt 없음, 레인보우 테이블 취약)
- 초기 비밀번호가 담당자코드(숫자 2자리)
- **조치:** `bcrypt` 또는 `argon2` 전환, 비밀번호 최소 길이 정책

### 1.5 SQL 인젝션 위험
- `resolution.py` 라인 48: 테이블명이 f-string으로 직접 삽입됨
- **조치:** 화이트리스트 검증 추가

### 1.6 파일 업로드 보안
- 확장자만 검증, 실제 MIME 타입 미확인
- **조치:** `python-magic`으로 실제 파일 타입 검증

### 1.7 프롬프트 인젝션
- `extraction.py`: 사용자 입력(`raw_text`)이 LLM 프롬프트에 직접 삽입
- 악의적 텍스트로 시스템 프롬프트 우회 가능
- **조치:** 입력 이스케이핑 + 시스템/사용자 프롬프트 경계 강화

## 2. 일반 개선사항

### 2.1 DB 커넥션 풀 부재
- `database.py`: 매 요청마다 새 연결 생성/해제
- **조치:** PostgreSQL은 `psycopg2.pool` 또는 `sqlalchemy` 커넥션 풀 적용

### 2.2 인덱스 부족
- `order_lines`, `match_candidates`, `feedback_log` 등에 인덱스 없음
- 데이터 증가 시 조회 성능 급락
- **조치:** 주요 검색 컬럼에 인덱스 추가

### 2.3 입력 검증 강화
- `orders.py`: `raw_text` 길이 제한 없음 (DoS 가능)
- `cust_code`, `cust_name` 형식 검증 없음
- **조치:** Pydantic Field로 min_length, max_length, regex 제약

### 2.4 에러 처리 표준화
- 에러 종류별 구분 없이 일괄 Exception 처리
- 사용자에게 복구 방법 안내 부족
- **조치:** 커스텀 예외 클래스 + 글로벌 예외 핸들러

### 2.5 로깅/모니터링
- 성능 메트릭 없음 (API 응답시간, LLM 토큰 사용량)
- 알림 시스템 없음 (동기화 실패, ERP 전송 실패)
- **조치:** structlog 도입, 핵심 작업에 시간측정 추가, 실패 시 알림

### 2.6 페이징
- 목록 API에 offset/페이징 없음
- 데이터 증가 시 전체 로딩으로 성능 저하
- **조치:** offset, sort_by, sort_order 파라미터 추가

### 2.7 보안 헤더
- X-Content-Type-Options, X-Frame-Options, HSTS 등 미설정
- **조치:** 보안 헤더 미들웨어 추가

## 3. AI/LLM 활용 강화 방안

### 3.1 프롬프트 캐싱 (비용 절감)
- Anthropic API의 `cache_control` 기능 활용
- 시스템 프롬프트를 캐싱하면 반복 호출 시 토큰 비용 ~90% 절감
```python
system=[{
    "type": "text",
    "text": SYSTEM_PROMPT,
    "cache_control": {"type": "ephemeral"}
}]
```

### 3.2 작업별 모델 분리 (비용/속도 최적화)
- **간단한 작업** (단순 텍스트 추출, 3줄 이하): Haiku → 빠르고 저렴
- **일반 작업** (발주서 파싱, 매칭): Sonnet → 현재 유지
- **복잡한 작업** (대량 매칭, 판단이 어려운 케이스): Opus → 높은 정확도

### 3.3 자동 학습 파이프라인
- 현재: 수동으로 학습 데이터 등록 → 수동 확인
- **개선:** ERP 전송 성공 시 자동으로 학습 데이터 축적
  - ERP 전송 완료 = 사용자가 최종 확인한 정확한 매칭
  - 이 데이터를 자동으로 po_training_pairs에 저장
  - few-shot 예시가 자연스럽게 축적

### 3.4 피드백 루프 강화
- 현재: feedback_log 테이블에 수정 이력 저장
- **개선:** 축적된 피드백을 분석하여 프롬프트 자동 개선
  - 자주 틀리는 패턴 분석 → 시스템 프롬프트에 주의사항 추가
  - 거래처별 특수 규칙 자동 학습

### 3.5 신뢰도 기반 자동처리 (STP 확대)
- 현재: CONFIDENCE_THRESHOLD=0.90 이상이면 자동처리
- **개선:** 거래처별 동적 임계값
  - 학습 데이터 많은 거래처: 임계값 낮춤 (0.85)
  - 신규 거래처: 임계값 높임 (0.95)
  - 자동처리율 점진적 향상

### 3.6 AI 품질 분석 대시보드
- 거래처별 AI 매칭 정확도 추적
- 시간별 STP(자동처리)율 그래프
- 자주 실패하는 상품/거래처 하이라이트
- 토큰 사용량 및 비용 추적

### 3.7 멀티모달 활용 확대
- 현재: 이미지 OCR에만 Vision 사용
- **확장 가능:**
  - 카카오톡 대화 스크린샷 → 주문 내용 자동 추출
  - 팩스 이미지 → 발주서 자동 처리
  - 손글씨 메모 → 주문 항목 인식

### 3.8 거래처별 맞춤 프롬프트
- 현재: 모든 거래처에 동일 프롬프트 사용
- **개선:** 거래처별 특성을 프롬프트에 반영
  - "이 거래처는 항상 박스 단위로 주문합니다"
  - "규격명에 색상이 포함되어 있습니다"
  - 축적된 학습 데이터에서 패턴 자동 추출

### 3.9 자연어 검색
- 현재: 모델명/키워드 기반 단가표 검색
- **개선:** Claude를 활용한 의미 기반 검색
  - "삼성 50인치 TV" → 정확한 모델코드 매칭
  - "지난주 주문한 거 다시" → 이전 주문 참조

### 3.10 일괄 처리 자동화
- 매일 특정 시간에 미처리 발주서 자동 수집
- AI가 자동으로 파싱 → 매칭 → 초안 생성
- 아침에 출근하면 확인만 하면 되는 워크플로우
